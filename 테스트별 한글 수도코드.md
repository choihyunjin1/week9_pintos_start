# Pintos Project 1 테스트별 한글 수도코드

이 문서는 `PROJECT 1: Threads`의 **실제 테스트 이름 기준**으로, 각 테스트가 무엇을 검증하는지, 어느 파일을 수정해야 하는지, 어떤 규칙을 만족해야 하는지, 그리고 코드를 쓰기 전에 어떤 **한글 수도코드**를 머리에 그려야 하는지를 정리한 문서입니다.

이 문서의 목표는 단순합니다.

> 테스트 하나를 보고  
> "이건 어느 파일의 어떤 규칙 문제인지"를 바로 판단하고,  
> 한글 수도코드를 먼저 적은 뒤,  
> 필요한 10줄씩 구현할 수 있게 돕는 것

같이 보면 좋은 문서:

- [개념학습과 테스트기반 구현 가이드.md](./개념학습과%20테스트기반%20구현%20가이드.md)
- [학습가이드 로드맵.md](./학습가이드%20로드맵.md)

## 1. 이 문서를 사용하는 방법

테스트 하나를 잡았을 때 아래 순서로 봅니다.

1. 테스트 이름을 찾습니다.
2. `이 테스트가 보는 핵심 규칙`을 읽습니다.
3. `주로 볼 파일`과 `주로 볼 함수`를 확인합니다.
4. `한글 수도코드`를 먼저 적고 나서 실제 C 코드로 옮깁니다.
5. `흔한 실패 원인`을 보고 edge case를 빠뜨리지 않았는지 확인합니다.

중요한 점:

- 이 문서는 **테스트 통과를 목표로 한 구현 안내서**입니다.
- 다만 실제 통과는 여전히 **세부 자료구조 설계, yield 시점, interrupt context 처리, fixed-point 계산 정확도**에 달려 있습니다.
- 그래서 각 테스트에 대해 "어떤 규칙을 빠뜨리면 깨지는지"를 최대한 적었습니다.

## 2. 공통 판단 프레임

테스트가 깨졌을 때 먼저 아래 다섯 가지 중 어디 문제인지 분류합니다.

| 문제 유형 | 대표 테스트 | 먼저 의심할 것 |
|---|---|---|
| 상태 전이 | `alarm-*` | `BLOCKED`, `READY`, wakeup |
| priority 정렬/선점 | `priority-preempt`, `priority-fifo` | ready list 정렬, yield 타이밍 |
| synchronization waiters | `priority-sema`, `priority-condvar` | waiter ordering |
| priority donation | `priority-donate-*` | donation 전파/복구 |
| 수식/주기 계산 | `mlfqs-*` | fixed-point, 4 tick/1초 갱신 |

## 3. Alarm 테스트

## 3-1. `alarm-single`

### 이 테스트가 보는 핵심 규칙

- 한 thread가 정확히 잠들고, 시간이 되면 깨어나야 합니다.

### 주로 볼 파일 / 함수

- `pintos/devices/timer.c`
- `timer_sleep()`
- `timer_interrupt()`

### 한글 수도코드

```text
timer_sleep(ticks):
    if ticks <= 0:
        return

    인터럽트를 잠깐 끈다
    현재 thread의 wakeup_tick = 현재 tick + ticks
    sleep_list에 현재 thread를 넣는다
    현재 thread를 BLOCKED 상태로 바꾼다
    인터럽트를 복구한다

timer_interrupt():
    현재 tick 증가
    깨울 시간이 지난 thread를 sleep_list에서 꺼내 READY로 바꾼다
```

### 통과 체크

- [ ] busy waiting이 없다
- [ ] 실제로 block 된다
- [ ] wakeup 후 ready list로 돌아간다

### 흔한 실패 원인

- `thread_yield()` 반복으로 sleep을 흉내 냄
- wakeup tick 계산 오류

---

## 3-2. `alarm-multiple`

### 핵심 규칙

- 여러 thread가 각자 다른 시각에 정확히 깨어나야 합니다.

### 한글 수도코드

```text
sleep_list를 wakeup_tick 기준으로 정렬한다

timer_interrupt():
    sleep_list 앞에서부터 보며
    wakeup_tick <= 현재 tick 인 thread를 모두 깨운다
```

### 흔한 실패 원인

- sleep_list 정렬이 안 됨
- 하나만 깨우고 끝남

---

## 3-3. `alarm-simultaneous`

### 핵심 규칙

- 같은 tick에 자야 하는 여러 thread를 모두 깨워야 합니다.

### 한글 수도코드

```text
while sleep_list가 비어있지 않고
      맨 앞 thread의 wakeup_tick <= 현재 tick:
    그 thread를 꺼내 unblock 한다
```

### 흔한 실패 원인

- `if`로 하나만 깨움
- 리스트 순회 중 remove 처리 오류

---

## 3-4. `alarm-zero`

### 핵심 규칙

- `ticks == 0`이면 잠들지 않고 바로 리턴합니다.

### 한글 수도코드

```text
if ticks <= 0:
    return
```

---

## 3-5. `alarm-negative`

### 핵심 규칙

- 음수 tick도 잠들지 않고 바로 리턴합니다.

### 한글 수도코드

```text
if ticks <= 0:
    return
```

---

## 3-6. `alarm-priority`

### 핵심 규칙

- 깨어난 뒤에도 priority scheduling 규칙이 유지되어야 합니다.

### 주로 볼 파일 / 함수

- `timer.c`
- `thread.c`
- `thread_unblock()`
- 필요하면 interrupt 종료 시 yield 예약

### 한글 수도코드

```text
timer_interrupt()에서 thread를 깨운 뒤
그 thread가 현재 thread보다 priority가 높다면
인터럽트가 끝난 뒤 CPU를 넘길 준비를 한다
```

### 흔한 실패 원인

- wakeup만 하고 선점을 안 함
- interrupt context에서 직접 잘못된 yield 호출

## 4. Priority Scheduling 테스트

## 4-1. `priority-preempt`

### 핵심 규칙

- 더 높은 priority thread가 READY가 되면 현재 thread는 CPU를 넘겨야 합니다.

### 주로 볼 파일 / 함수

- `thread.c`
- `thread_unblock()`
- `thread_create()`

### 한글 수도코드

```text
thread가 READY 상태가 될 때:
    ready_list에 넣는다
    if 새 thread priority > current priority:
        yield 또는 interrupt-return yield를 준비한다
```

### 흔한 실패 원인

- ready list에는 넣지만 current가 계속 실행됨

---

## 4-2. `priority-change`

### 핵심 규칙

- current priority를 바꿨을 때 더 높은 ready thread가 있으면 yield 해야 합니다.

### 한글 수도코드

```text
thread_set_priority(new_priority):
    base priority를 바꾼다
    effective priority를 다시 계산한다
    if ready_list에 나보다 높은 thread가 있으면:
        yield 한다
```

### 흔한 실패 원인

- 값만 바꾸고 yield 안 함

---

## 4-3. `priority-fifo`

### 핵심 규칙

- 같은 priority끼리는 순서가 불필요하게 깨지지 않아야 합니다.

### 한글 수도코드

```text
ready_list 정렬 시
같은 priority에 대해서는 기존 순서를 최대한 유지한다
```

### 흔한 실패 원인

- comparator가 동일 priority도 뒤집음

---

## 4-4. `priority-sema`

### 핵심 규칙

- semaphore waiters 중 가장 높은 priority가 먼저 깨어나야 합니다.

### 주로 볼 함수

- `sema_down()`
- `sema_up()`

### 한글 수도코드

```text
sema_down():
    semaphore가 0이면
    current를 waiters에 priority 순으로 넣고 block 한다

sema_up():
    waiters가 있으면
    가장 높은 priority waiter를 꺼내 unblock 한다
```

### 흔한 실패 원인

- FIFO로만 처리

---

## 4-5. `priority-condvar`

### 핵심 규칙

- condition variable도 highest priority waiter를 먼저 깨워야 합니다.

### 주로 볼 함수

- `cond_wait()`
- `cond_signal()`

### 한글 수도코드

```text
cond_wait():
    semaphore wrapper를 만든다
    condition waiters에 넣는다
    lock을 풀고 semaphore down 한다
    깨어나면 다시 lock을 잡는다

cond_signal():
    가장 높은 priority thread를 품고 있는 waiter를 골라
    그 waiter의 semaphore를 up 한다
```

### 흔한 실패 원인

- wrapper 안의 thread priority를 비교하지 않음

---

## 5. Priority Donation 테스트

## 5-1. `priority-donate-one`

### 핵심 규칙

- 높은 priority thread가 lock을 기다리면 holder가 donation을 받아야 합니다.

### 주로 볼 파일 / 함수

- `synch.c`
- `thread.h`
- `lock_acquire()`
- `lock_release()`

### 한글 수도코드

```text
lock_acquire(lock):
    if holder가 있고 current priority가 더 높으면:
        holder effective priority를 current priority까지 올린다
    lock을 획득한다

lock_release(lock):
    이 lock 관련 donation을 제거하고 priority를 복구한다
```

---

## 5-2. `priority-donate-multiple`

### 핵심 규칙

- 여러 donor 중 최고 priority가 반영되어야 합니다.

### 한글 수도코드

```text
thread는 donor 목록을 가진다
effective priority = max(base priority, 모든 donor priority)
```

### 흔한 실패 원인

- 마지막 donor만 반영

---

## 5-3. `priority-donate-multiple2`

### 핵심 규칙

- donation이 여러 개 들어오고 빠질 때 복구가 정확해야 합니다.

### 한글 수도코드

```text
lock_release(lock):
    이 lock 때문에 들어온 donor만 제거한다
    남은 donor와 base priority 중 최대값으로 다시 계산한다
```

### 흔한 실패 원인

- donor 전체를 비워버림

---

## 5-4. `priority-donate-nest`

### 핵심 규칙

- donation이 한 단계가 아니라 chain으로 전파되어야 합니다.

### 한글 수도코드

```text
holder = current가 기다리는 lock의 holder
while holder가 존재:
    holder priority를 donor priority 이상으로 올린다
    if holder도 다른 lock을 기다리면:
        holder = 그 lock의 holder
    else:
        break
```

---

## 5-5. `priority-donate-chain`

### 핵심 규칙

- 긴 chain도 같은 방식으로 계속 전파되어야 합니다.

### 한글 수도코드

```text
donation 전파는 한 번 하고 끝나는 것이 아니라
waiting lock chain을 따라 반복된다
```

### 흔한 실패 원인

- 한 단계만 donation 처리

---

## 5-6. `priority-donate-sema`

### 핵심 규칙

- holder가 다른 이유로 block되어도 donation 상태는 유지되어야 합니다.

### 한글 수도코드

```text
donation은 holder가 RUNNING일 때만 아니라
lock을 들고 있는 동안 유지된다
```

---

## 5-7. `priority-donate-lower`

### 핵심 규칙

- donation 중 base priority를 낮춰도 effective priority는 donor가 있는 동안 유지됩니다.

### 한글 수도코드

```text
thread_set_priority(new_priority):
    base priority만 바꾼다
    effective priority는 donor 포함 최대값으로 다시 계산한다
```

## 6. MLFQS 테스트

## 6-1. `mlfqs-load-1`

### 핵심 규칙

- `load_avg`를 1초마다 공식대로 갱신해야 합니다.

### 한글 수도코드

```text
매 1초마다:
    ready_threads 수를 구한다
    idle thread는 제외한다
    load_avg를 공식대로 계산한다
```

---

## 6-2. `mlfqs-load-60`

### 핵심 규칙

- 많은 thread에서도 ready thread 수와 `load_avg` 계산이 맞아야 합니다.

### 한글 수도코드

```text
ready_threads 수 계산 시
현재 running thread가 idle이 아니면 포함한다
```

---

## 6-3. `mlfqs-load-avg`

### 핵심 규칙

- 장시간에 걸쳐 `load_avg` 추세가 공식과 맞아야 합니다.

### 한글 수도코드

```text
load_avg는 매 tick이 아니라 매 1초마다만 갱신한다
```

---

## 6-4. `mlfqs-recent-1`

### 핵심 규칙

- `recent_cpu` 계산식과 갱신 주기가 정확해야 합니다.

### 한글 수도코드

```text
매 tick마다:
    current가 idle이 아니면 recent_cpu += 1

매 1초마다:
    모든 thread recent_cpu를 공식대로 다시 계산한다
```

---

## 6-5. `mlfqs-fair-2`

### 핵심 규칙

- 2개 thread가 공정하게 CPU를 나눠 가져야 합니다.

### 한글 수도코드

```text
매 4 tick마다:
    모든 thread priority를 다시 계산한다
```

---

## 6-6. `mlfqs-fair-20`

### 핵심 규칙

- 많은 thread에서도 fairness가 유지되어야 합니다.

### 한글 수도코드

```text
priority 재계산 후
ready_list가 새 priority 순서를 반영하게 유지한다
```

---

## 6-7. `mlfqs-nice-2`

### 핵심 규칙

- nice 값 차이가 priority에 반영되어야 합니다.

### 한글 수도코드

```text
thread_set_nice(new_nice):
    current nice를 바꾸고
    priority를 다시 계산하고
    필요하면 yield 한다
```

---

## 6-8. `mlfqs-nice-10`

### 핵심 규칙

- 다양한 nice 값에 대해 priority 계산이 안정적으로 맞아야 합니다.

### 한글 수도코드

```text
priority = PRI_MAX - (recent_cpu / 4) - (nice * 2)
결과는 범위를 넘지 않게 clamp 한다
```

---

## 6-9. `mlfqs-block`

### 핵심 규칙

- blocked thread도 전체 thread 목록 갱신 대상에 포함되어야 합니다.

### 한글 수도코드

```text
매 1초/4tick 갱신 때:
    running만 아니라 모든 thread를 순회해 갱신한다
```

이 문서의 목적은 "정답 코드를 대신 써주는 것"이 아니라, **각 테스트를 구현 가능한 수준의 하위 수도코드로 내려 주는 것**입니다.

즉, 이 문서까지 준비되면 이제 해야 할 일은 명확합니다.

> 테스트 하나를 고르고  
> 관련 함수 몇 개만 열고  
> 수도코드를 실제 C 코드로 옮기고  
> 그 테스트 하나를 다시 돌린다.
